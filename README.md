# Warehouse DQN

A Deep Q-Network (DQN) agent that learns to navigate a randomly generated
10x10 warehouse grid, pick up a package, and deliver it to a drop-off point
— trained entirely from scratch with PyTorch.

## Overview

The agent controls a robot on a grid containing walls, a package, and a
delivery point. On each step it can move up, down, left, or right. Its goal
is to reach the package, pick it up, and carry it to the delivery point
while avoiding walls and doing so in as few steps as possible.

The warehouse layout, robot position, package position, and delivery
position are randomized each episode (guaranteed solvable via a BFS
reachability check), so a fully generalized policy needs to learn to
navigate *arbitrary* layouts, not memorize a fixed map.

## Usage

### Train

```bash
python train.py
```

This runs `config.EPISODES` episodes, printing progress every 100 episodes:

```
Episode:            500 / 5000
Average Reward:     236.85
Best Reward:        236.88
Average Loss:       0.1141
Epsilon:            0.082
Pickup Rate (100):  100.0%
Success Rate (100): 100.0%
Replay Size:        50000
```

It saves:
- `models/best_model.pth` — the checkpoint with the best rolling 100-episode
  average reward seen so far
- `models/latest_model.pth` — updated every 500 episodes
- `models/final_model.pth` — the model at the end of training
- `plots/reward.png`, `plots/loss.png`, `plots/epsilon.png`,
  `plots/success_rate.png` — training curves
- `plots/*.npy` — raw history arrays, so plots can be regenerated without
  retraining

If you want to regenerate the plots later (e.g. after tweaking a plot's
style) without rerunning training, use the saved history arrays directly:

```bash
python utils/visualization.py
```

This reads `plots/*.npy` and rewrites `plots/*.png`.

### Evaluate

Watch a trained agent solve fresh warehouses, acting greedily (no random
exploration):

```bash
python evaluate.py --model models/best_model.pth --episodes 5
```

Useful flags:
- `--episodes N` — number of episodes to run
- `--no-render` — skip printing the grid every step (much faster; good for
  getting a quick success-rate estimate over many episodes)
- `--delay 0.3` — seconds to pause between rendered steps (for
  screen-recording a demo)

## The environment

**State**: the warehouse is encoded as a flattened 100-element vector
(10x10), where each cell is one of:

| Value | Meaning |
|---|---|
| 0 | Empty |
| 1 | Wall |
| 2 | Robot |
| 3 | Package (only shown if not yet picked up) |
| 4 | Delivery point |

**Actions**: `0` = up, `1` = down, `2` = left, `3` = right.

**Reward shaping**:

| Event | Reward |
|---|---|
| Regular move | -1 |
| Hitting a wall / boundary | -5 |
| Getting closer to the current goal (package or delivery) | +1 |
| Getting farther from the current goal | -1 |
| Picking up the package | +50 |
| Delivering the package | +200 |
| Getting stuck (10 consecutive invalid moves) | -50, episode ends |

The distance-based shaping reward gives the agent a training signal on
*every* step, rather than only at pickup/delivery, which would otherwise
be many steps apart and a very sparse learning signal.

The stuck-detection rule ends an episode early if the agent repeatedly
bumps into walls/boundaries, rather than letting it burn the rest of
`MAX_STEPS` accumulating penalties — this was important for training
stability (see [Notes on training](#notes-on-training) below).

## The algorithm

Standard DQN, with a few stability techniques layered on:

- **Experience replay** (`agent/replay_buffer.py`) — transitions are stored
  and sampled in random mini-batches, breaking the correlation between
  consecutive steps.
- **Target network** (`agent/dqn_agent.py`) — a separate, periodically
  synced copy of the Q-network is used to compute Bellman targets, which
  keeps training from chasing a constantly-moving target. The target
  network is updated every `config.TARGET_UPDATE` *training steps* (not
  episodes — episode length varies too much for an episode-based schedule
  to be consistent).
- **Epsilon-greedy exploration** — epsilon decays each episode from
  `EPSILON_START` to `EPSILON_END`.
- **Huber loss** (`SmoothL1Loss`) instead of MSE — much less sensitive to
  the occasional large-magnitude target, which matters here because
  Q-values can span a wide range.
- **Gradient clipping** — caps the gradient norm so no single batch can
  destabilize the network.

## Curriculum learning

Training an agent to generalize across *fully random* maps from the start
turned out to be too hard a starting point (see notes below). `config.py`
supports training on a small, fixed pool of maps first:

```python
NUM_FIXED_MAPS = 1     # train on a single fixed map
NUM_FIXED_MAPS = 5      # a small pool
NUM_FIXED_MAPS = 20     # a larger pool
NUM_FIXED_MAPS = None   # fully random map every episode
```

The idea: verify the DQN can learn navigation at all on a fixed map, then
gradually widen the pool of maps it has to handle, before finally testing
it on fully random, never-seen-before layouts.

## Notes on training

A few things that came up during development, worth knowing if you're
tuning this further:

- **Reward scale matters more than it seems.** An early version let the
  agent accumulate wall-bump penalties for the full length of an episode
  (up to `MAX_STEPS`), which produced runaway negative rewards and unstable
  Q-value estimates. Capping this with early episode termination on
  repeated invalid moves fixed it.
- **Loss stabilizing doesn't mean the policy is good.** At one point loss
  was low and stable, but success rate was still stuck near 0% — the
  network had converged, just to a bad policy. This turned out to be a
  task-difficulty problem (fully random maps every episode), not an
  algorithm bug — confirmed by testing on a single fixed map, which reached
  100% success within a few hundred episodes.
- **Track more than just reward.** Adding a separate "pickup rate" metric
  (did the agent ever reach the package, regardless of full delivery) made
  it much easier to tell whether a failure was a navigation problem or a
  delivery-specific problem.

## Results

On a single fixed warehouse map, the agent reaches **100% success rate**
and **100% pickup rate**, holding both consistently for thousands of
episodes, with loss converging to a small, stable value. This validates
the DQN implementation itself.

Generalization across a wider pool of maps (5, then 20, then fully random)
is the current in-progress milestone — results will be updated here as
that training completes.

## Future improvements

- Complete the curriculum: 5 maps → 20 maps → fully random
- Try a CNN-based state encoder instead of a flattened MLP input, so the
  network can more directly learn local spatial patterns (walls, corridors,
  dead ends)
- Double DQN / Dueling DQN architectures
- Prioritized experience replay
- Optimizer/epsilon checkpointing to support resuming interrupted training
  runs (already supported by `DQNAgent.save`/`load`, just not wired into
  `train.py` yet)
